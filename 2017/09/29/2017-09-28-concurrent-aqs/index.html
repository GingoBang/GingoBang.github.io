Concurrent包：AbstractQueuedSynchronizer基础同步器

Posted by ruijie on 2017-09-29

AbstractQueuedSynchronizer是一个同步器的基础框架

一. AQS介绍

基于FIFO队列向外部提供一个基础框架，可以用于实现阻塞锁和一些同步器，比如semaphores等。其内部通过state: int变量来标识同步状态state=0表示没有线程占用共享资源的锁，state=N表示有线程占用共享资源的锁。

AQS支持两种模式：独占模式和共享模式，默认是独占模式。简单的说如果是独占模式，那么一个锁只能被一个线程操作，其他线程需要在同步队列中等待，共享模式下可以多个线程操作同一个锁，AQS可以同时支持这两种模式，不过一般情况下，实现类只支持某一种模式，ReadWriteLock则是同时使用这两种模式。

AQS中有两个重要的内部类，ConditionObject和Node，另外如果要基于AQS实现自己的同步器，可以通过使用getState(), setStates(), compareAndSetState()来操作state字段，另外按需要实现以下方法：

  • tryAcquire()
  • tryRelease()
  • tryAcquireShared()
  • tryReleaseShared()
  • isHeldExclusively()

上面的waitStatus可以为以下值：

  • 0: 值为0时表示节点处于同步队列中，等待获取锁
  • CANCELLED :值为1，在同步队列中的线程超时或者被中断，需要从同步队列中去掉这个Node节点，此时其waitStatus==CANCELLED，即结束状态，进入此状态后不会再改变
  • SIGNAL:值为-1，一个Node节点的后继节点被block了(park())，那么当前节点在准备释放或者cancel时需要去unpark它的后继节点，为了避免竞争，首先一个结点需要将其前驱节点置为SIGNAL，即可被唤醒状态，然后才执行park操作，防止阻塞
  • CONDITION:值为-2，当前节点目前处于condition队列中，在被transfer之前，都不能作为同步队列中的节点，即当前线程目前处于等待状态中，调用了Condition的signal()方法后将其转移到同步队列中
  • PROPAGATE:值为-3，在共享模式中，该状态标识结点的线程处于可运行状态

三. 内部类ConditionObject

ConditionObject是Condition接口的一个实现类，在具体看ConditionObject之前我们先看一下Condition接口

1. Condition接口

Object对象提供了一组监视器方法wait(),notify(),nodifyAll()，通过使用这些方法，可以实现多个线程之间的通信，比如经典的生产者-消费者模型，通过notify()方法可以唤醒阻塞在这个监视器上的随机一个线程。和监视器(synchronized)不同的是，通过对对象锁上的多个线程使用多个Condition对象，我们可以进行更细致的控制，精细的控制线程的休眠和唤醒操作。

2. ConditionObject的实现

上面介绍Node类时说到它维护了一个同步队列，是一个由Node对象组成的双向链表。而在ConditionObject中维护的是一个FIFO的单向链表，称为等待队列，每一个ConditionObject对应一个等待队列，一个AQS可以存在多个等待队列，但是只能有一个同步队列。等待队列中的元素同样是Node，封装的是线程引用，并且处于等待队列中的Node.waitStatus只有CANCELED和CONDITION两种状态。

通过调用Condition对象的await()方法将线程封装为Node对象，加入到等待队列中并且挂起，直到线程被唤醒，中断。被唤醒的线程将进入自旋开始尝试获取锁。线程的唤醒是由signal()方法实现的，它会将等待队列中的头结点移动到同步队列中，并且唤醒同步队列中的前驱节点，将其waitStatus置为SIGNAL。下面看一下是如何实现这些操作的。

这是整个signal()方法的实现流程，首先去判断是否为独占模式(需要子类自己实现)，如果不是独占模式直接抛出异常，之后如果条件队列的头结点不为空，执行唤醒过程，也就是从等待队列的头部开始唤醒(添加时是向尾部添加)，首先将尝试CAS更新头结点的waitStatus为0，即初始状态，如果CAS更新失败意味着当前节点的状态不为Node.CONDITION，这种情况只会出现在状态被更新为了CANCEL，因为不存在线程竞争，毕竟调用这个方法的线程是持有锁的，返回错误。

CAS更新出错后，会从头节点向下遍历循环，并把这些错误节点从条件队列里移除，直到成功，否则将firstWaiter，lastWaiter都更新为null。

CAS更新成功后，会调用Node的enq()方法，将节点移动到同步队列中，添加成功后返回节点的前驱节点，这时候将判断此前驱节点的waitStatus值，如果表明前驱节点被cancel了或者CAS更新其状态为Node.SIGNAL时失败了，将直接唤醒新添加节点的线程。成功更新的话会将前驱结点的状态更新为Node.SIGNAL,表明后继结点需要被唤醒。

除了signal()方法外，还提供了signalAll()方法，很简单，遍历条件队列执行signal()方法

2.2. await()方法

简单的说，await()需要将当前线程封装，添加到等待队列，释放线程占用的锁并且挂起，直到被唤醒或被中断。被唤醒后需要重新获取锁，
